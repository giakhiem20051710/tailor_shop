Giải Mã: 2-Phase Inventory Reservation & LUA Script

Bí kíp chống "bán lỗ", chống "cháy hàng" trong các đợt Flash Sale / Bán vé siêu tốc.

Vấn đề thực tế (Bài toán "Giành giật áo vest")

Bối cảnh: Tailor Shop mở bán Flash Sale 1 cái áo vest "Limited Edition" giá giảm 90% siêu rẻ. Trong kho MySQL hiện tại stock = 1.

Sự kiện: Đúng 12:00:00, cùng lúc có 10,000 khách truy cập bấm nút "Cho vào giỏ hàng" cùng một mili-giây.

Thảm họa (Nếu không dùng 2-Phase & Redis): Cả 10,000 luồng (threads) đều lao vào chọc Database: "Ê áo còn không?". Database kiểm tra thấy stock = 1, nên trả lời 10,000 người là: "Còn, mua đi". Kết quả: Hệ thống bán ra 10,000 cái áo dù trong kho chả có cái nào. Bạn đền tiền sấp mặt. Hơn nữa, với 10,000 kết nối MySQL cùng lúc, Database sẽ nổ tung (Quá tải).

Sơ đồ 2-Phase Reservation (Giữ chỗ 2 giai đoạn)

Giải pháp: Đem số lượng hàng bỏ lên RAM (Redis) để phản xạ cực nhanh. MySQL chỉ được đụng tới khi khách đã trả tiền thật.

Phase 1: Giai đoạn giữ chỗ (Giật Cúp)

10,000 User
Bấm Mua
DECR flash_sale:stock (Trừ 1)
REDIS CACHE
Chỉ 1 người được kết quả >= 0 (Thành công).
9,999 người bị trả về < 0 (Hết hàng).

Phase 2: Giai đoạn kết toán (Chồng Bạc)

1 User May Mắn
Thanh toán VNPay
Thanh toán OK -> Update Stock
MySQL DATABASE
Lúc này mới vào Database trừ kho thật:
UPDATE stock = stock - 1

Tại sao phải dùng LUA SCRIPT thay vì Code Java bình thường?

Nhiều người hỏi: "Tại sao không dùng Java để hỏi Redis: Số lượng còn bao nhiêu, nếu còn > 0 thì gọi lệnh Trừ 1 đi?".

Nguyên nhân là do Race Condition (Cuộc đua kỳ thú). Giữa khoảnh khắc bạn "Hỏi" và bạn "Trừ", thằng khác đã chen ngang!

1. Cách SAI (Dùng Code Java thường)

// 2 Luồng code chạy song song (Thread 1 và Thread 2) // Kho đang có 1 áo. // Bước 1: GET số lượng T1: int soLuong = redis.get("stock"); // KQ: 1 T2: int soLuong = redis.get("stock"); // KQ: 1 // Bước 2: Kiểm tra (Cả 2 đều thấy đúng) T1: if (soLuong > 0) { T2: if (soLuong > 0) { // Bước 3: Trừ đi T1: redis.decr("stock"); // Kho thành 0 T2: redis.decr("stock"); // Kho thành -1 lố cmnr! }

Kết quả: 2 người cùng trừ kho thành công trong khi chỉ có 1 áo. Lỗi nặng!

2. Cách CHUẨN (Dùng LUA Script)

-- Đây là đoạn code Lua chạy nằm nguyên cục BÊN TRONG thẻ nhớ Redis local stock = tonumber(redis.call('get', KEYS[1])) if (stock > 0) then redis.call('decr', KEYS[1]) return 1 -- Thành công (Cho rớt vô giỏ hàng) else return 0 -- Thất bại (Báo hết hàng) end

ĐIỀU KỲ DIỆU CỦA LUA: Redis chạy trên 1 luồng (Single Thread). Khi nó đang chạy 1 đoạn Lua Script, NÓ KHÓA CHẶT MỌI THỨ KHÁC (Atomicity). Không ai được phép chọc vào kho cho đến khi cục Code Lua này chạy xong từ trên xuống dưới. 100% không bao giờ bán âm kho.

Câu Trả Lời "Ăn Điểm" Khi Phỏng Vấn (Tóm Tắt)

"Em dùng cơ chế 2-Phase để tránh chết Database. Phase 1 (Mua liền): Em chặn hàng chục ngàn Request bằng cách trừ tồn kho tạm trên RAM (Redis) bằng lệnh DECR. Vì Redis chịu tải tới 100,000 req/s nên nó gánh vác rất nhẹ nhàng. Ngay lập tức từ chối 9,999 người hết hàng, chỉ đẩy đúng 1 người có hàng vô giỏ."

"Sau đấy, Phase 2 (Thanh toán): Khi cái 1 người may mắn kia chuyển tiền VNPay thành công, em mới dùng luồng chạy ngầm chậm rãi (như Kafka hoặc cronjob/sync) chọc vào MySQL để UPDATE stock = stock - 1 thực sự trên đĩa cứng. Nên MySQL không bao giờ sập."

"Đặc biệt ở Phase 1, em bắt buộc bọc logic kiểm tra và trừ kho vào một LUA Script. Vì nếu dùng Java gọi `GET` rồi gọi `DECR` rời rạc, giữa 2 lệnh đó sẽ xảy ra Race Condition gây bán lố hàng. Redis thực thi Lua Script mang tính Nguyên tủ (Atomic) - nghĩa là khoá toàn bộ cục script lại chạy trong đúng 1 tick, không request nào chen ngang được, dập tắt hoàn toàn lỗi Overselling (Bán lố)."
