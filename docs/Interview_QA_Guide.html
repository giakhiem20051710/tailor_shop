Q1 Mô tả kiến trúc tổng thể của project? ▼

Q2 Tại sao chọn Modular Monolith thay vì Microservices? ▼

Project có 16 domain modules nhưng tất cả chạy trong 1 JVM. Lý do:

  1. Team nhỏ — deployment đơn giản hơn, giảm operational overhead (không cần service mesh, distributed tracing phức tạp)
  2. Clean boundaries — mỗi module có domain/repo/service/dto/controller riêng biệt, sẵn sàng tách ra
  3. Event-driven loose coupling — modules communicate qua Spring Events, dễ dàng chuyển sang message broker (Kafka/RabbitMQ) khi cần
  4. Dễ scale đọc ngang — có thể tách module thành microservice sau nếu load tăng
💡

Key insight: Monolith-first approach — build monolith có boundaries tốt, rồi extract services khi thực sự cần. Tránh premature optimization.

II. Security & Authentication

3 câu hỏi
Q4 Giải thích flow authentication? ▼
  1. User POST /api/v1/auth/login với username/password
  2. Backend validate bằng BCrypt, tạo access token (30 phút, HMAC SHA-256) + refresh token (7 ngày)
  3. Token chứa claims: sub (userId), roles, type (access/refresh), jti (UUID cho revocation)
  4. Client gửi Authorization: Bearer <token> mỗi request
  5. JwtAuthFilter (OncePerRequestFilter) validate token → load CustomUserDetails → set SecurityContext
  6. Khi access token hết hạn, client gọi /auth/refresh với refresh token để lấy token mới
Client ──POST──▶ /auth/login {username, password} ◀────── {accessToken (30min), refreshToken (7d)} Client ──GET───▶ /api/v1/products Header: Authorization: Bearer eyJhbG... ◀────── 200 OK + data // Token expired: Client ──POST──▶ /auth/refresh {refreshToken} ◀────── {newAccessToken, newRefreshToken}
Q5 Có bao nhiêu role? Role-based access hoạt động thế nào? ▼

4 roles:

ADMIN STAFF TAILOR CUSTOMER

Backend: Controller dùng @PreAuthorize("hasRole('ADMIN')")

Frontend: Dùng ProtectedRoute + RoleGuard components với các preset:

GuardRoles cho phép

Q6 Security headers nào được cấu hình? ▼

Spring Security:

  • CSRF disabled (stateless JWT, không cần CSRF protection)
  • CORS configurable từ env — CORS_ALLOWED_ORIGINS
  • X-XSS-Protection: 1; mode=block
  • X-Frame-Options: DENY
  • X-Content-Type-Options: nosniff
  • Cache-Control: no-cache, no-store

Nginx (Frontend): thêm X-Frame-Options: SAMEORIGIN, gzip compression, static asset cache 1 year (immutable).

Q8 Rate limiting implementation chi tiết? ▼

Custom RateLimitFilter (extends OncePerRequestFilter):

  1. Extract client IP từ X-Forwarded-For / X-Real-IP headers (hỗ trợ reverse proxy)
  2. Check IP whitelist (127.0.0.1, localhost) — skip nếu whitelisted
  3. Redis key: IP-based counter với TTL 1 phút
  4. Increment counter, check limits: 100 req/min, 10 req/sec
  5. Set response headers: X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset
  6. Vượt limit → trả 429 Too Many Requests
🛡️

Graceful degradation: Nếu Redis down → allow all traffic (fail-open strategy). Đảm bảo service availability.

IV. Flash Sale & Concurrency

2 câu hỏi
Q9 Flash Sale xử lý high-concurrency thế nào? ⭐ HOT ▼

Dùng SERIALIZABLE transaction isolation + Pessimistic Locking:

  1. LOCK flash sale row — SELECT ... FOR UPDATE via @Lock(PESSIMISTIC_WRITE) trên findByIdForUpdate()
  2. Validate — sale active, not expired, not sold out, not cancelled
  3. Check minimum purchase quantity (minPurchase)
  4. Check available stock — totalQuantity - soldQuantity - reservedQuantity
  5. Check per-user limit — sum of previous purchases against maxPerUser
  6. Create reservation — hold 10 phút (status ACTIVE, có expiresAt)
  7. Atomic update — increment reservedQuantity trên flash sale entity
  8. Create order — status PENDING_PAYMENT, deadline 10 phút
@Transactional(isolation = Isolation.SERIALIZABLE) public PurchaseResult purchaseFlashSale(Long flashSaleId, int quantity, Long userId) { // Step 1: Pessimistic lock FlashSaleEntity sale = repo.findByIdForUpdate(flashSaleId); // Step 2-5: Business validations validateSale(sale); checkAvailableStock(sale, quantity); checkUserLimit(sale, userId); // Step 6-7: Create reservation + update stock FlashSaleReservation reservation = createReservation(sale, quantity, userId); sale.setReservedQuantity(sale.getReservedQuantity() + quantity); // Step 8: Create order FlashSaleOrder order = createOrder(sale, reservation, userId); return new PurchaseResult(order, reservation); }

Scheduled Jobs:

  • Auto-activate: SCHEDULED → ACTIVE khi đến startTime
  • Auto-end: close sale khi hết endTime
  • Release expired reservations — trả lại reservedQuantity
Q10 Reservation Pattern là gì? Tại sao cần? ▼

Reservation Pattern = giữ hàng tạm thời trước khi thanh toán thực sự. Giải quyết vấn đề: user chọn mua nhưng chưa thanh toán → cần "hold" stock để người khác không mua mất.

Purchase ──▶ Reservation (ACTIVE, 10 min) ──▶ Payment ──▶ Confirm │ timeout 10 min ▼ EXPIRED → Release Stock

Entity FlashSaleReservation có business methods:

  • isExpired() — check if expiresAt đã qua
  • isValid() — active + not expired
  • getRemainingSeconds() — thời gian còn lại

Nếu không pay trong 10 phút → reservation expired → release quantity → sản phẩm available lại.

Q12 Dead-Letter Queue (DLQ) hoạt động thế nào? ⭐ HOT ▼

Custom DLQ infrastructure gồm 4 components:

  1. @RetryableEvent annotation — đánh dấu event handlers cần retry. Config: maxRetries (default 3), saveToDlq, exception filters
  2. RetryableEventAspect — AOP @Around advice, bọc handler với exponential backoff (base 100ms, max 5s, ±20% jitter)
  3. EventRetryService — persist failures vào failed_events table, scheduled retry mỗi 60s
  4. DlqAdminController — Admin API: /api/v1/admin/dlq — xem stats, retry/skip dead events
Event Handler ──▶ @RetryableEvent (AOP intercept) │ ├── Success ──▶ Done ✅ │ └── Fail ──▶ Retry (exponential backoff, max 3x) │ ├── Retry success ──▶ Done ✅ │ └── All retries fail ──▶ Save to DLQ (failed_events table) │ Status flow: PENDING → PROCESSING → PROCESSED | DEAD | SKIPPED │ Scheduled job ──▶ Retry every 60s │ Admin API ──▶ Manual retry/skip

Q14 Có những relationship patterns nào đặc biệt trong DB design? ▼

1. Polymorphic Entity (Generic Type + ID)

Cart và Favorites không dùng FK cứng mà dùng itemType + itemId:

// CartItem có thể chứa Product HOẶC Fabric @Enumerated(EnumType.STRING) private CartItemType itemType; // PRODUCT | FABRIC private Long itemId; // Favorite có thể chứa Product, Fabric, HOẶC ImageAsset private FavoriteItemType itemType; // PRODUCT | FABRIC | IMAGE_ASSET private Long itemId;

2. Versioned Entity (Full History)

Measurements dùng groupId + version + isLatest để track lịch sử đo body:

private String groupId; // Nhóm các versions lại private Integer version; // 1, 2, 3... private Boolean isLatest; // Chỉ 1 record là latest

3. Self-referential (Parent-Child)

SeasonalChallenge có parent → subChallenges (tree structure cho challenges lồng nhau).

Q16 Bulk image upload flow hoạt động thế nào? ▼
  1. Request pre-signed URLs — client gọi POST /bulk-upload/presigned-urls → nhận list S3 pre-signed URLs
  2. Direct upload — client upload trực tiếp lên S3 (bypass server — giảm bandwidth)
  3. Submit job — POST /bulk-upload/submit với list S3 keys → tạo BulkUploadJobEntity (status PENDING, UUID jobId)
  4. Async processing — backend download từ S3 → Gemini AI analyze → generate thumbnail (300×300) + large (1200px width) → persist ImageAssetEntity
  5. Poll progress — client poll GET /jobs/{jobId} xem: totalFiles, processedFiles, successCount, failedCount
  6. Auto-cleanup — job expires sau 24h
💡

Key insight: Pre-signed URLs cho phép client upload trực tiếp lên S3 mà không cần proxy qua server → giảm latency và server load đáng kể cho batch 200+ images.

XII. Câu hỏi nâng cao

3 câu hỏi
Q28 Nếu Redis down, hệ thống xử lý thế nào? (Graceful Degradation) ▼

Graceful degradation — hệ thống vẫn hoạt động khi Redis unavailable:

FeatureBehavior khi Redis down
Rate LimitingFail-open: allow all traffic. Service availability > security
Caching@ConditionalOnProperty — cache miss → query DB trực tiếp. Slower but functional
Flash SaleVẫn hoạt động qua DB pessimistic locking. Redis chỉ hỗ trợ, không critical
💡

Implementation: @ConditionalOnExpression check S3/Redis credentials khi startup. Nếu thiếu → dùng mock implementation hoặc bypass.

Q29 Làm sao scale hệ thống khi traffic tăng? ▼

4 chiến lược scale:

  1. Horizontal (Microservices) — tách modules thành services riêng. Event-driven đã sẵn → chỉ cần đổi transport từ in-memory events sang message broker (Kafka/RabbitMQ)
  2. Vertical — tăng resource limits trong docker-compose.prod.yml (CPU, memory)
  3. Database — MySQL read replicas cho read-heavy queries, Redis cluster cho cache distribution
  4. CDN + Caching — Cloudflare CDN (đã config), static assets cache immutable 1 year, Redis cache với tiered TTL
Q30 Giải thích flow tạo Order qua Wizard (Saga-like)? ▼

Saga-like orchestration — 1 API call POST /orders/wizard coordinate across 4 modules:

POST /orders/wizard │ ├── Step 1: Create Order │ (customer, products, note, expected budget) │ ├── Step 2: Create/Link Measurement │ (body measurements for customer) │ ├── Step 3: Create Appointment │ (fitting schedule, available slots check) │ └── Step 4: Auto-generate Invoice (subtotal, tax, discount calculation) │ ▼ Events Published: ├── OrderCreatedEvent → reserve fabric inventory ├── Send notification to customer └── Send notification to staff/admin

Q32 Khó khăn lớn nhất khi build project này? Cách giải quyết? ▼

Gợi ý trả lời:

1. Flash Sale Concurrency — Race condition khi nhiều user mua cùng lúc. Giải quyết bằng pessimistic locking + reservation pattern + scheduled cleanup. Đã test với multiple concurrent requests — stock consistency đảm bảo 100%.

2. Event Reliability — Event handler fail dẫn đến data inconsistency. Giải quyết bằng custom DLQ với @RetryableEvent + exponential backoff + dead letter persistence + admin retry API.

3. AI Integration Stability — Gemini API rate limits và intermittent failures. Giải quyết bằng 3-retry strategy + AIRateLimiter + async batch processing + progress tracking.

4. Cross-module Communication — Tight coupling giữa order/billing/appointment/measurement. Giải quyết bằng event-driven architecture — modules communicate qua events, không gọi trực tiếp.

Q33 Nếu có thêm thời gian, sẽ cải thiện gì? ▼

Gợi ý trả lời (từ technical debt → innovation):

  1. Message Broker — Migrate từ Spring Events (in-memory) sang Kafka/RabbitMQ cho reliable event delivery cross-service
  2. Test Coverage — Tăng unit test coverage (services + repositories), integration tests cho critical flows (flash sale, order wizard)
  3. CQRS — Implement Command Query Responsibility Segregation cho read-heavy endpoints (product catalog, trend analysis)
  4. OAuth2/OpenID Connect — Migrate custom JWT sang Spring Authorization Server hoặc Keycloak cho SSO, social login
  5. API Versioning — Formal versioning strategy (URL path hoặc header-based) thay vì chỉ /api/v1
  6. Kubernetes — Migrate từ Docker Compose sang K8s cho auto-scaling, self-healing, rolling updates

Q34 Tại sao chọn Java 21 thay vì Java 17? ❓ WHY ▼

Java 21 là LTS mới nhất (Sep 2023), được chọn vì:

  1. Long-term support — LTS cycle dài hơn Java 17, nhận security patch đến ~2031. Đầu tư cho tương lai.
  2. Record types — Project dùng record cho DTOs (UserRequestDTO, NotificationResponse, DlqStats). Records giảm boilerplate: auto-generated equals(), hashCode(), toString().
  3. Virtual Threads ready — Java 21 có Virtual Threads (Project Loom) GA. Dù chưa dùng, khi load tăng có thể enable ngay mà không cần refactor async code.
  4. Spring Boot 3.3 alignment — Spring Boot 3.3.x có first-class Java 21 support, auto-configuration cho Virtual Threads.
  5. Zero migration cost — Bắt đầu project mới với Java 21 không mất chi phí migration. Nếu dùng 17 rồi upgrade sau sẽ pain hơn.
💡

Trade-off: Một số library cũ (JavaFaker) dùng reflection nặng, có thể gặp compatibility issues. Nhưng lợi ích LTS + future-proofing outweigh risks.

Q35 Tại sao dùng HMAC SHA-256 cho JWT thay vì RSA? ❓ WHY ▼

JWT signing dùng symmetric HMAC-SHA256 (Keys.hmacShaKeyFor(secretBytes)):

  1. Single-service architecture — Cả token issuer lẫn validator đều nằm trong cùng 1 Spring Boot app. Không cần asymmetric key vì không có service riêng chỉ verify mà không sign.
  2. Đơn giản — Chỉ cần 1 env var JWT_SECRET (≥32 bytes). RSA cần generate key pair, lưu trữ, rotation, JWKS endpoint.
  3. Performance — HMAC-SHA256 nhanh 10-20x so với RSA-256. Quan trọng vì mỗi API request đều phải validate token.
  4. Configuration tối thiểu — 1 biến JWT_SECRET trong .env vs quản lý public/private key files qua nhiều deployment.
🔄

Khi nào chuyển RSA? Khi tách sang microservices — auth service sign token, các service khác chỉ verify bằng public key. Hiện tại monolith → HMAC là optimal.

Q36 Tại sao chọn BCrypt cho password hashing? ❓ WHY ▼

new BCryptPasswordEncoder() trong SecurityConfig:

  1. Spring Security default — BCrypt là recommended password encoder trong Spring ecosystem. Không cần thêm dependency.
  2. Adaptive cost factor — Work factor có thể tăng khi hardware mạnh hơn (default strength 10 ≈ ~100ms per hash).

  4. Battle-tested — Dùng production hàng thập kỷ, security properties được hiểu rõ.

Q37 Tại sao chọn Pessimistic Locking thay vì Optimistic Locking cho Flash Sale? ❓ WHY ▼

Flash sale dùng @Lock(PESSIMISTIC_WRITE) + SERIALIZABLE isolation:

  1. High contention — Hàng trăm/ngàn user mua cùng lúc trên cùng 1 sản phẩm. Optimistic locking (@Version) sẽ gây massive retry storms vì hầu hết transaction đều conflict.
  2. Overselling = thảm họa — Bán quá stock là nghiêm trọng. SELECT ... FOR UPDATE đảm bảo không có 2 transaction đọc/sửa stock cùng lúc.
  3. Đơn giản, chính xác — Lock row → validate → update → release. Không cần retry logic phức tạp như Optimistic.
  4. MySQL InnoDB row-level lock — InnoDB hỗ trợ row-level locking native. Lock chỉ 1 row flash sale, không block toàn table.
Optimistic Locking (BAD for flash sale) Thread A: READ stock=10, version=1 Thread B: READ stock=10, version=1 Thread A: UPDATE stock=9, version=2 ✅ Thread B: UPDATE stock=9, version=2 ❌ // OptimisticLockException → RETRY // 1000 users → 999 retries → cascade failures Pessimistic Locking (GOOD for flash sale) Thread A: SELECT ... FOR UPDATE → LOCKS row Thread B: WAITS for lock... Thread A: UPDATE stock=9, COMMIT → releases lock Thread B: Gets lock → READ stock=9 → UPDATE stock=8 ✅ // Serialized, no retries, correct every time
⚖️

Trade-off: Throughput thấp hơn (serialize transactions), nhưng correctness > throughput cho flash sale. Normal product purchase có thể dùng optimistic locking.

⚠️

Q39 Tại sao chọn Tailwind CSS thay vì Material UI / styled-components? ❓ WHY ▼

Project dùng Tailwind CSS 3.4 — không có thêm component library nào:

  5. Không component lock-in — Không phụ thuộc library component, tự do custom mọi thứ.
🎨

Trade-off: Verbose class names, không có pre-built components (nhưng đây là intentional cho luxury brand). Custom keyframes/animations defined trong tailwind.config.js: fade-in-up, shimmer, pulse-glow.

Q40 Tại sao tự viết HTTP client (fetch) thay vì dùng Axios? ❓ WHY ▼

Custom HttpClient class 325 dòng wrap native fetch API:

  1. Zero dependencies — fetch native trong tất cả modern browsers + Node 18+. Bỏ ~13KB (gzipped) Axios dependency.
  2. Full control — Custom interceptor chain, retry logic, timeout (AbortController), auto auth token injection. Tất cả tailored cho app's specific needs.
  3. Bundle size matters — Project có Three.js (~600KB) + MediaPipe (~2MB) cho 3D/AR. Giảm dependency count ở chỗ có thể.
  4. Modern API — fetch + AbortController là standard cách handle request cancellation. Axios vẫn wrap XMLHttpRequest (legacy).
  5. API giống Axios — api.get(), api.post(), api.put(), api.delete() — team quen thuộc.
⚙️

Trade-off: 325 dòng code maintain (vs import Axios). Phải tự implement JSON parsing, error extraction, FormData handling. fetch không reject trên HTTP error status — phải handle trong handleResponse.

Q42 Tại sao chọn STOMP/SockJS thay vì plain WebSocket? ❓ WHY ▼

WebSocket config: @EnableWebSocketMessageBroker + STOMP protocol + SockJS fallback:

  1. Message routing native — STOMP cung cấp /topic (broadcast) và /queue (point-to-point) destinations. User-specific notifications: /user/{id}/queue/notifications — routing tự động.
  2. SockJS fallback — Transparent long-polling fallback khi WebSocket bị block bởi corporate firewalls hoặc proxies.
  3. Spring integration — spring-boot-starter-websocket hỗ trợ STOMP first-class: @MessageMapping, SimpMessagingTemplate, user destination resolution.
  4. Heartbeats — STOMP protocol có heartbeat negotiation (10s incoming/outgoing), tự động phát hiện dead connections.
  5. Auto-reconnect — @stomp/stompjs Client handles reconnection delay 5s + fallback polling mỗi 2 phút.
🔌

Trade-off: STOMP frames thêm header overhead lên WebSocket frames. Thêm ~30KB dependencies (@stomp/stompjs + sockjs-client). Plain WebSocket nhẹ hơn nhưng phải tự implement routing/subscription.

Q44 Tại sao Flyway thay vì Liquibase? Và tại sao hiện tại disable Flyway? ❓ WHY ▼

Tại sao Flyway:

  1. Plain SQL — Migrations là file .sql thuần. Project chỉ dùng MySQL → không cần Liquibase's XML/YAML abstraction.
  2. Versioned sequential — V1__init.sql → V25__gamification_tables.sql. 25 migrations cho thấy schema evolution rõ ràng.
  3. Developer-friendly — DBA và dev viết SQL trực tiếp, không cần học thêm DSL.
  4. Spring Boot auto-config — Zero-config integration: chỉ cần dependency + migration files trên classpath.

Tại sao disable (flyway.enabled: false)?

  1. Development speed — Hibernate ddl-auto: update tự sync schema khi thay đổi entities. Không cần viết migration file cho mỗi thay đổi nhỏ.
  2. Dual strategy — Dev dùng ddl-auto: update, production enable Flyway cho controlled migrations. 25 migration files đã sẵn sàng.
🗄️

Trade-off: Liquibase có rollback generation, database-agnostic changesets. Nhưng project MySQL-only → SQL migrations đủ. No rollback migrations (U__) = forward-only strategy.

Q45 Tại sao dùng multi-stage Docker builds? ❓ WHY ▼

Cả Backend và Frontend Dockerfile đều 2 stages:

Backend Dockerfile Stage 1: maven:3.9-eclipse-temurin-21 (builder) → mvn dependency:go-offline (cache layer) → mvn package → JAR file Stage 2: eclipse-temurin:21-jre-jammy (runtime) → Copy JAR only → non-root user Frontend Dockerfile Stage 1: node:20-alpine (builder) → npm ci → vite build → static files Stage 2: nginx:alpine (serving) → Copy dist/ only → custom nginx.conf
  1. Image size — JDK ~800MB vs JRE ~200MB. node_modules ~500MB+ vs nginx + static ~30MB. Multi-stage loại bỏ build tools khỏi runtime.
  2. Security — Runtime images không chứa compilers, package managers, source code. Non-root user cho backend. Attack surface nhỏ nhất.
  3. Cache optimization — pom.xml/package.json copy trước cho dependency caching. Source changes không invalidate dependency layer.
  4. Separation of concerns — Build environment (Maven/Node) decoupled khỏi runtime (JRE/Nginx).
Q46 Tại sao chọn FastAPI (Python) cho Try-On service thay vì tích hợp vào Java backend? ❓ WHY ▼

  2. GPU isolation — ML inference cần GPU (CUDA). Tách thành microservice riêng → backend chính (JVM) không bị ảnh hưởng bởi GPU memory.
  3. Independent scaling — Try-on service scale riêng biệt (thêm GPU instances) mà không scale toàn bộ backend.
  4. FastAPI async — async def + ASGI (uvicorn) cho concurrent image upload/processing. StreamingResponse trả ảnh trực tiếp (no base64 overhead).
  5. Auto-documentation — FastAPI tự generate OpenAPI/Swagger docs. Pydantic validation cho request/response models.
🐍

Q47 Tại sao dùng event-driven (Spring Events) thay vì gọi trực tiếp giữa services? ❓ WHY ▼

Modules communicate qua ApplicationEventPublisher.publishEvent():

  1. Module decoupling — Order module publish OrderCreatedEvent; notification, invoice, appointment modules react independently. Không circular dependencies.
  2. Async processing — @Async("eventExecutor") offload side effects (email, invoice) sang background threads. API response trả về ngay.
  3. Transaction safety — @TransactionalEventListener(AFTER_COMMIT) đảm bảo listeners chỉ fire nếu main transaction thành công. Không phantom notifications cho rolled-back orders.
  4. Fault tolerance — @RetryableEvent + DLQ — failed events được retry exponential backoff + persist cho manual intervention.
  5. Không cần message broker — Spring ApplicationEventPublisher là in-process. Không cần setup RabbitMQ/Kafka cho monolith.
Direct Call (BAD — tight coupling) OrderService → NotificationService.send() → InvoiceService.create() → AppointmentService.schedule() // OrderService phụ thuộc 3 services → circular dependency risk Event-Driven (GOOD — loose coupling) OrderService → publish(OrderCreatedEvent) NotificationListener → send notification @Async InvoiceListener → create invoice @Async AppointmentListener → schedule @Async // OrderService không biết ai consume event

Q50 Tại sao Redis dùng @ConditionalOnProperty? Tại sao không bắt buộc? ❓ WHY ▼

RedisConfig và CacheConfig đều có @ConditionalOnProperty(name = "spring.data.redis.host", matchIfMissing = false):

  1. Graceful degradation — App chạy được không cần Redis. Nếu spring.data.redis.host không set → RedisConfig/CacheConfig beans không tạo → fallback về no-cache.
  2. Developer experience — Dev mới clone repo → chạy ./mvnw spring-boot:run không cần Docker/Redis. Caching là performance optimization, không phải functional requirement.
  3. Testing — Unit/integration tests không cần Redis running. Tránh ConnectionRefusedException khi test.
  4. Docker-compose flexibility — local-db.yml kết nối host MySQL nhưng vẫn chạy Redis container. Running không compose → skip Redis.
🛡️

Trade-off: Không Redis = không caching + không rate limiting. App chậm hơn và unprotected against abuse trong dev mode. matchIfMissing = false = default "no Redis" → cần explicit opt-in.

Q51 Tại sao dùng polymorphic Cart thay vì separate cart tables? ❓ WHY ▼

CartItemEntity dùng generic reference: CartItemType itemType (enum: PRODUCT, FABRIC, SERVICE) + Long itemId:

  1. Single cart table — 1 bảng cart_items xử lý tất cả item types. Không cần cart_products, cart_fabrics, cart_services. Schema đơn giản, query đơn giản.
  2. Business domain fit — Tailor shop bán vải (theo mét), sản phẩm may sẵn, và dịch vụ (may đo, sửa đồ). Khách thêm tất cả vào một giỏ hàng.
  3. Unified checkout — 1 giỏ hàng → 1 checkout flow dù mix product types.
  4. Extensibility — Thêm item type mới: (1) thêm enum value, (2) thêm switch case trong validate/enrich. Không cần schema migration.
⚠️

Trade-off: Không có FK constraint từ item_id đến specific table (vì references nhiều tables). Referential integrity phải enforce bằng application code. switch statements ở nhiều nơi — mỗi type mới cần update nhiều chỗ.

Q52 Tại sao CSRF disabled? Có an toàn không? ❓ WHY ▼

SecurityConfig: csrf.disable() — CÓ an toàn trong context này:

  1. Stateless JWT — Authentication qua Authorization: Bearer header. Browser không tự động gửi header này trong cross-site requests (khác cookie). CSRF attack vector không tồn tại.
  2. Không dùng cookies cho auth — CSRF exploit dựa trên browser tự gửi session cookies. Project dùng JWT trong header → không vulnerable.
  3. REST API standard — Stateless REST APIs thường disable CSRF. Spring Security docs cũng recommend disable khi dùng non-browser clients hoặc JWT.
  4. SPA architecture — React frontend gửi requests qua fetch() với explicit headers. Không phải traditional form submission.
🔐

Khi nào cần enable CSRF? Nếu chuyển sang cookie-based sessions (httpOnly cookies cho JWT), hoặc hỗ trợ traditional form submissions. Hiện tại = an toàn.

Q53 Tại sao rate limiter dùng fail-open (allow all) khi Redis down? ❓ WHY ▼

RateLimitFilter — khi Redis unavailable → allow all traffic:

  1. Availability > Protection — Business priority: khách hàng phải luôn mua được hàng. Redis down (hiếm) → chấp nhận temporarily unprotected hơn là block tất cả traffic.
  2. Fail-open vs Fail-closed — Fail-closed (block all) khi Redis down = DDoS chính mình. Toàn bộ legitimate users bị block.
  3. Redis down = temporary — Redis restart nhanh (seconds). Giai đoạn unprotected rất ngắn.
  4. Defense in depth — Rate limiting không phải layer bảo vệ duy nhất. Nginx, Cloudflare cũng có rate limiting ở edge.
Fail-Open (Chosen) ✅ Redis DOWN → Allow all requests → App vẫn hoạt động Risk: Temporary unprotected (seconds) Mitigation: Edge rate limiting (Nginx/Cloudflare) Fail-Closed (NOT chosen) ❌ Redis DOWN → Block ALL requests → 503 cho mọi user Risk: Self-inflicted outage Impact: Revenue loss, customer frustration
