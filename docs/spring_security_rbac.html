Giải Nghĩa: Phân Quyền Vai Trò (RBAC)
& Lớp Lọc An Ninh (Filter Layer)

Bí kíp thiết kế tường lửa bảo vệ Ứng dụng Backend (Spring Boot)

1. RBAC là gì? (Role-Based Access Control)

Đây là cơ chế "Phân quyền dựa trên chức danh". Trong Tailor Shop, bạn có 3 chức danh chính:

  • CUSTOMER (Khách): Chỉ được đổi mật khẩu, xem lịch sử mua hàng của bản thân, xem sản phẩm.
  • TAILOR (Thợ may): Được xem danh sách đơn may của chính mình, cập nhật tiến độ may. Không được đụng vào hóa đơn thanh toán.
  • ADMIN/STAFF: Quyền sinh sát tối thượng. Được sửa giá, ban user, xem tổng doanh thu...

Bất cứ ai gõ vào cửa Backend API (Controller) đều phải chìa thẻ căn cước (Tấm vé JWT Token) ra. Trên thẻ phải ghi rõ chức danh (Role).

2. Cuộc Chiến: Chặn ở Cổng Cảng (Filter) hay Chặn ở Từng Phòng (@PreAuthorize)

Internet / Hacker

Cầm thẻ giả, hoặc thẻ hết hạn, hoặc thẻ Customer đòi vào phòng Admin.

JwtAuthenticationFilter (Cổng Thành)
Config: /admin/** -> Chỉ ADMIN
Tước thẻ giả ngay đây!
Controller (Các Căn Phòng)
Phòng `deleteUser()`
Phòng `getTailorOrders()`
Phòng `getProducts()` (Mở)

Vấn đề của Cách 1: Chặn Ở Từng Phòng (Dùng tag @PreAuthorize)

Đây là cách các lập trình viên mới vào nghề hay làm: Viết ra 100 cái API (100 căn phòng), phòng nào quan trọng thì dán cái bùa @PreAuthorize("hasRole('ADMIN')") trước cửa phòng đó.

Dùng @PreAuthorize (Rủi ro vỡ trận)
@RestController @RequestMapping("/api/admin/users") public class AdminUserController { @PreAuthorize("hasRole('ADMIN')") @DeleteMapping("/{id}") public void deleteUser() { ... } // CHẾT DỞ!! Dev mới vào công ty // code vội quá QUÊN dán bùa ở đây! // -> Hacker (Customer) xông thẳng vào // gõ API này và chôm sạch thu nhập!! @GetMapping("/revenue") public Data getDoanhThu() { ... } }
Dùng Filter Chain (Phòng Thủ Trùng Điệp)
// Trong hàm SecurityFilterChain (Cổng Thành) // Mọi Request đi qua mạng đều BẮT BUỘC // phải lọt qua cái trạm gác này trước! http.authorizeHttpRequests(auth -> auth // Khai báo tính tổng quát cao (Enforce tập trung) .requestMatchers("/api/admin/**").hasRole("ADMIN") .requestMatchers("/api/tailor/**").hasRole("TAILOR") // Phải khóa hết toàn bộ phòng còn lại .anyRequest().authenticated() ); // -> Lính canh ở Cổng Cảng sẽ đạp văng bất cứ // ai không có thẻ ADMIN mà dám bén mảng tới // nguyên khu vực /api/admin/* // Dev bên trong CÓ QUÊN dán bùa cũng chả sao!

🔑 Giải thích Keyword "Enforce Tập Trung":
Có nghĩa là bạn vạch ra quy tắc pháp luật (Policies) ở một chỗ duy nhất (file SecurityConfig). Nếu sếp yêu cầu đổi luật: "Cho thêm ông Manager vào xem Doanh Thu", bạn chỉ cần mở ĐÚNG 1 FILE cổng thành ra sửa. Bạn không bao giờ lo cái rủi ro "Bỏ sót, quên gài pass" (Security bypass) do một cậu Dev Junior nào đó trong team vô tình code API mới mà quên gắn tag bảo vệ.

Dịch sang Tiếng Người để Trả Lời Phỏng Vấn:

Dịch đoạn văn mẫu của bạn:

"Dạ thưa anh, hệ thống của em thiết lập Cổng Cảng (SecurityFilterChain) ngay từ bên ngoài. Mọi gói tin (Request) bắn tới Server đều bị chặn lại ở trạm gác JwtAuthenticationFilter. Ở trạm này mổ cái thư (Token JWT) ra để xét xem người này cầm Căn cước gì (Role gì), rồi lưu vào não bộ của Server (Set SecurityContextHolder)."

"Lý do em dùng Cổng Cảng (Filter) để phân quyền thay vì rải lính canh ở từng căn phòng (xài tag @PreAuthorize trên từng Method) là vì bài toán Thắt Chặt Tập Trung (Enforce). Code cấu hình ở Filter cho phép em tuyên bố thẳng: Mọi thằng đi vào khu vực /api/admin/ đều phải có tag ADMIN. Cấm ngoại lệ!"

"Nếu em xài @PreAuthorize rải rác trong code Controller, lỡ sau này dự án mở rộng lên 500 cái Controller, dự án tuyển thêm thằng Dev mới (em nói dối xíu là dự án có team 🤣), thằng Dev lơ ngơ code cái hàm xóa Database mà QUÊN dán dòng @PreAuthorize lên đầu hàm, thế là cả ngàn lớp bảo vệ đứt gãy, Hacker (Bypass) xông thẳng vào xóa cạn Data. Làm ở Filter thì không bao giờ để lọt lỗi NGỚ NGẨN (Human Error) này!"
