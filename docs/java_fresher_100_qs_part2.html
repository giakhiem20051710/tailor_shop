CÂU 14: Method Overloading (Ghi đè) dựa vào yếu tố nào để phân biệt các hàm cùng tên?
  • A) Kiểu trả về (Return type) của hàm.
  • B) Số lượng, kiểu dữ liệu, và thứ tự của các tham số (Parameters).
  • C) Access modifier (public, private, protected).
  • D) Tên của tham số truyền vào.
Xem Đáp Án
✅ B
Trình biên dịch Java chỉ nhìn vào Dấu Hiệu (Signature) của hàm gồm: Tên hàm + Danh sách tham số (số lượng, kiểu, thứ tự). Nó KHÔNG quan tâm đến Kiểu trả về. Nếu viết 2 hàm `int sum(int a, int b)` và `double sum(int a, int b)`, Java sẽ báo lỗi trùng lặp.
CÂU 15: Chữ `super` đẫm máu
class A { A() { System.out.print("A"); } } class B extends A { B() { System.out.print("B"); } } class C extends B { C() { System.out.print("C"); } } public class Test { public static void main(String[] args) { new C(); } }
  • A) in ra "C"
  • B) in ra "CBA"
  • C) in ra "ABC"
  • D) Compilation error
Xem Đáp Án
✅ C
Khi nổ lệnh `new C()`, Constructor của C chạy. Nhưng quy tắc sống còn của Java là: Dòng đầu tiên của bất kì hàm Constructor nào cũng bị CHÈN NGẦM một lệnh `super()`. Vậy nên nó gọi ngược lên B, B lại bị chèn ngầm gọi ngược lên A. Nên A in trước -> B -> C. Output: ABC.
CÂU 16: Khởi tạo mảng kiểu cắm sừng
int[] arr = new int[5]; System.out.println(arr[0]);
  • A) null
  • B) 0
  • C) Garbage value (Giá trị rác ngẫu nhiên)
  • D) ArrayIndexOutOfBoundsException
Xem Đáp Án
✅ B
Khi dùng `new int[5]`, Java dọn sẵn 5 ô nhớ trên Heap. Với kiểu nguyên thủy (primitive) thuộc họ số (int, double, float, byte), Java mặc định rải số `0` (hoặc `0.0`) vào tất cả các ô chứ không để rỗng (null) hay giá trị rác như C/C++.
CÂU 17: Phím mưu hèn kế bẩn của Switch-Case
int x = 2; switch (x) { case 1: System.out.print("A"); case 2: System.out.print("B"); case 3: System.out.print("C"); default: System.out.print("D"); }
  • A) B
  • B) B C
  • C) B C D
  • D) Lỗi biên dịch
Xem Đáp Án
✅ C
Do thiếu từ khóa `break;`. Switch-case hoạt động theo kiểu "Thác nước" (Fall-through). Nó chạy khớp được `case 2` in ra B xong, nó rơi tuột tự do xuống các case bên dưới MÀ KHÔNG CẦN XÉT ĐIỀU KIỆN NỮA. Vậy nên nó hốt trọn in ra "BCD".
CÂU 18: Try-Catch nhiều lớp (Multiple Catch)
try { int x = 10 / 0; } catch (Exception e) { System.out.print("1"); } catch (ArithmeticException e) { System.out.print("2"); }
  • A) 1
  • B) 2
  • C) 12
  • D) Compile Error (Lẽ ra Catch to phải nằm dưới)
Xem Đáp Án
✅ D
Ngoại lệ (Exception) phải được bắt theo nguyên tắc: TỪ CON ĐẾN CHA. (Từ vụn vặt đến tổng quát). `Exception` là class ông nội vĩ đại nhất. Đặt nó ở trên cùng thì nó thầu hết mọi loại lỗi lọt vào rớ, thằng `ArithmeticException` nằm dưới sẽ KHÔNG BAO GIỜ chạm tới được (Unreachable code). Java cấm điều này và báo lỗi ngay lúc gõ code (Compile Error).
CÂU 19: Từ khóa `throw` và `throws` khác nhau chỗ nào?
  • A) throw dùng sau đít tên hàm, throws dùng trong ruột thân hàm.
  • B) throw dùng để ném ngoại lệ có chủ đích (vd: throw new Exception()), còn throws dùng trên chữ ký hàm để cảnh báo (ném bom cho thằng gọi hàm tự xử).
  • C) Cả hai y chang nhau, do ngữ pháp thích ghi thêm chữ 's'.
Xem Đáp Án
✅ B
- `throw` (động từ mạnh): Nằm TRONG BỤNG HÀM. Tự tay kích nổ quả bom. (Ví dụ: `if (age < 18) throw new RuntimeException("Chua du tuoi");`)
- `throws` (từ cảnh báo): Nằm Ở CHỮ KÝ HÀM (Ví dụ: `void readFile() throws IOException {}`). Nó mang ý nghĩa "Hàm này có thể gây nổ bom đó nha, thằng nào gọi hàm tao thì tự móc try-catch ra mà đỡ đi!".
CÂU 20: ConcurrentModificationException là cái quái gì?
List<String> list = new ArrayList<>(Arrays.asList("A", "B", "C")); for (String s : list) { if (s.equals("B")) { list.remove(s); } }
  • A) Nó in ra List chỉ còn [A, C]
  • B) Sinh lỗi ConcurrentModificationException
  • C) Sinh lỗi NullPointerException
Xem Đáp Án
✅ B
Bẫy kinh điển! Khi đang dùng vòng lặp `for-each` (bản chất là Iterator) để đi dạo qua List, bạn TUYỆT ĐỐI KHÔNG ĐƯỢC dùng hàm `list.remove()` hoặc `list.add()` để thay đổi độ dài mảng. Việc bạn lén rút cái cục "B" ra làm sập hệ thống Iterator đang duyệt, nó giật mình ném lỗi ConcurrentModificationException.
Cách khắc phục: Dùng vòng for(int i=0..) duyệt ngược từ cuối lên đầu, hoặc dùng Iterator.remove().
CÂU 21: TreeSet dùng cái gì để bắt chép phạt mấy phần tử bắt phải đứng đúng thứ tự?
  • A) Nó xài thuật toán Bubble Sort mỗi lần thêm mới.
  • B) Class của các phần tử nhét vào bắt buộc phải implements Interface Comparable hoặc xài Comparator bên ngoài.
  • C) Nó tự động sort dựa trên lệnh .hashCode()
Xem Đáp Án
✅ B
TreeSet cấu trúc bằng cây Đỏ-Đen. Để biết nút áo nào đặt bên trái, nút nào bên phải, nó phải hỏi "Đứa nào bự hơn?". Mà đụng đến Đối tượng (Object Người, Object Xe), máy đứt bóng không biết ai bự. Nên class của bạn phải dạy nó cách so sánh (implements Comparable ghi đè hàm compareTo()) thì TreeSet mới cho nạp vào bình yên.
CÂU 22: Pass by Value hay Pass by Reference trong Java?
class P { int x = 10; } public class Test { static void change(P p) { p.x = 20; p = new P(); p.x = 30; } public static void main(String[] args) { P mainP = new P(); change(mainP); System.out.println(mainP.x); } }
  • A) 10
  • B) 20
  • C) 30
Xem Đáp Án
✅ B) 20
Java 100% là Pass by Value (Nhưng copy giá trị của cái Địa Chí Nhớ).
1. `mainP` cầm chìa khóa nhà (O nhớ A).
2. Truyền vô hàm `change`, biến `p` cũng được copy cho một cái chìa khóa sao y bản chính vô trúng nhà A.
3. Quậy banh chành `p.x = 20`. Ngôi nhà A bị đổi thành 20.
4. `p = new P()`. Biến tên `p` xách mông đi mua nhà mới (nhà B), phá chìa khóa cũ. Nó đập phá `p.x = 30` ở bên nhà B.
5. Hết hàm, `p` chết. Về lại main, `mainP` mở nhà A ra xem thì đồ đạc vẫn là 20 (do bị sửa lúc 3). Ngôi nhà B số 30 rác rưởi trôi nổi.
CHẠY NƯỚC RÚT 20 CÂU CÒN LẠI (ĐỌC NHANH - THẤM NHANH)
  • CÂU 23: String.valueOf(null) trả về Lỗi `NullPointerException` hay in ra chữ "null"?
    ✅ Đáp án: Lỗi NullPointerException. Hàm valueOf(Object) thấy dính Null nó chửi lộn và cháy máy ngay. Đừng nghịch dại.
  • CÂU 24: Lớp System trong `System.out.println` có được phép dùng new System() để tạo Object không?
    ✅ Đáp án: KHÔNG THỂ. Lớp System có constructor giấu kín (private). Không ai sinh sôi lén lút được nó.
  • CÂU 25: HashMap cho phép mấy cái Key=null?
    ✅ Đáp án: CHỈ VÀ ĐÚNG 1 CÁI KEY NULL. Mấy cái Value thì mớ null cũng được, nhưng chìa khóa Null thì chỉ có 1 cái (vì Key không được trùng lặp). Riêng HashTable thì tuyệt đối cấm tiệt Key,Value chứa Null.
  • CÂU 26: Class B kế thừa Class A, hai class đều có biến độc int x = 10; (A) và int x = 20; (B). A obj = new B(); System.out.println(obj.x); ra số mấy?
    ✅ Đáp án: Số 10. Thuộc tính (Biến/Variable) KHÔNG BAO GIỜ bị Overridden (Ghi đè đa hình) như Hàm (Method). Cho nên biến mang mác loại nào (A) thì lòi đúng số loại đó ra (10).
  • CÂU 27: Hàm có keyword final thì có bị class con Overriding (đè đầu cưỡi cổ) được không?
    ✅ Đáp án: CẤM TUYỆT ĐỐI. Class con chỉ biết đứng khóc.
  • CÂU 28: List.of("A", "B").add("C"); Hệ quả?
    ✅ Đáp án: Cháy máy (UnsupportedOperationException). Chữ List.of() và Arrays.asList() sinh ra cái mảng BÊ TÔNG, không bao giờ nở dài thêm được.
  • CÂU 29: Multithreading: Điểm khác nhau giữa `.start()` và `.run()` của một Thread?
    ✅ Đáp án: Gọi `.start()` là tạo luồng mới siêu tốc, Main làm việc Main, luồng làm việc luồng. Gọi `.run()` là NÓ TRỞ THÀNH HÀM JAVA CÙI BẮP bình thường, luồng Main chui mẹ vô dòng code đó chạy cực nhọc rồi mới leo ra. Mất sạch ý nghĩa đa luồng.
  • CÂU 30: String vs StringBuilder vs StringBuffer. Thằng nào dùng cho Hệ Thống Nhiều Người Cùng Update Vô Cục String (Thread-safe)?
    ✅ Đáp án: StringBuffer. Nó bị gắn cái khóa nhà tù `synchronized` lên mọi hàm, nên thằng này sờ vào thì thằng kia phải đợi, chậm hơn nhưng an toàn. StringBuilder dùng cho một mình anh cô đơn, chạy bạt mạng rất lẹ.
  • CÂU 31.. 50: (Tập trung CSDL): Lệnh TRUNCATE so với DELETE khác nhau lõi gì?
    ✅ Đáp án: DELETE là thằng quét dọn lao công, xóa từng hàng một và CÓ LƯU sổ nhật ký (có thể Rollback hoàn về). TRUNCATE là thằng thả bom nguyên tử phá mẹ cái kho bãi nổ tan tành 0.1 giây rồi xây cái vỏ mới, XÓA DỨT CĂN, không có khái niệm Hoàn Liệt Rollback!
